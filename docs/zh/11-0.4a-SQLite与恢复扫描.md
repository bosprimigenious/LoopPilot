## 0.4-a 是什么

**0.4-a（状态可靠）** 是 0.4 个人日用闭环的第一个子阶段，交付：

- SQLite 状态存储的生产路径（migration、备份、校验）
- `loop-pilot db` 四命令 + `recovery-scan`
- `doctor` 在 sqlite 后端下的扩展检查

**不包含** inbox/queue/today（0.4-b）、审阅 CLI（0.4-c）、每日总结与调度（0.4-d）。

## 为何默认仍是 json

仓库默认 `config/loop-pilot.yaml` 保持：

```yaml
runtime:
  state_backend: json
```

原因：

1. **0.1 CI 基线不退化** — 现有 GitHub Actions 与 fixture 跑法依赖 json 后端
2. **0.4-a 是 opt-in** — 个人验收时在本地或专用 config 目录启用 sqlite
3. **渐进迁移** — 0.4-b/c/d 业务表在 sqlite 稳定后再叠加上去

启用 0.4-a 时在 config 中设置：

```yaml
runtime:
  state_backend: sqlite
  sqlite_path: var/state/loop_pilot.db
  lock_dir: var/locks
```

推荐用独立 config 目录，不改动默认 config：

```bash
loop-pilot --config-dir var/acceptance-0.4a-config db status
```

| 模块 | 路径 | 作用 |
|------|------|------|
| migrations | `src/loop_pilot/storage/migrations.py` | schema v1、增量 migration |

## recovery-scan 行为

`recovery-scan` **只扫描、给建议，不自动恢复**。

| 情况 | 分类 | 建议动作 |
|------|------|----------|
|  phase = ACTING | `acting_interrupted` | **`manual_review_required`** — 不自动 resume |
| WAITING_APPROVAL | `waiting_approval` | `needs_human` |
| TERMINATED + FAILED | `failed_run` | `review_or_resume` |
| 超时无进展 | `stale_run` | `manual_review_required` |
| lock 文件残留 | `stale_lock` | `blocked` |

**关键规则：** ACTING 阶段被中断时，系统标记需人工审阅，**不会**自动 resume。正式 resume/approve  CLI 在 **0.4-c**。

## 与 0.4-b/c/d 的边界

```text
0.4-a 状态可靠 ──► 0.4-b 任务入口 ──► 0.4-c 审阅 ──► 0.4-d 总结与调度
     (本页)            inbox/queue          review CLI       summary/schedule
```

| 子阶段 | 0.4-a 已做 | 留给后续 |
|--------|-----------|----------|
| **0.4-b** | SQLite + migration 就绪 | `inbox add/list`、`queue promote`、`today` |
| **0.4-c** | recovery-scan 可见待处理 run | `review list`、`approve/reject/defer/cancel`、`resume` |
| **0.4-d** | — | `summary today/week`、`schedule print/install --dry-run` |

不可跳过 0.4-a：0.4-b 的 `inbox_items` 等表依赖已稳定的 migration 机制。
