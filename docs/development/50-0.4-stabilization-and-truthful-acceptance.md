### Phase 0.6 — Canonical manifest and deferred review (Codex P1/P2)

**Manifest sole-writer principle**

- Only `finalize_terminal_artifacts()` in `src/loop_pilot/runtime/terminal_artifacts.py` may write the final `artifact-manifest.json`.
- Loop implementations (InternLoop, PaperLoop, DailyNewsLoop) must **not** emit a legacy manifest write after terminalization.
- InternLoop patch runs call `mark_patch_run_waiting_review()` → `write_review_suggestion()` → `finalize_terminal_artifacts()`; the returned in-memory `ArtifactManifest` is informational only.
- The finalizer scans `run_dir` for on-disk files, preserves prior `kind` / `human_readable` metadata when present, **recomputes every sha256 from disk**, excludes `artifact-manifest.json` from its own listing, and atomically writes the manifest last.

**Patch review phase semantics (Codex P2-1 / P2-2 / P2-3)**

- Before human approve, patch runs (`patch.diff` present, no decision): `phase=WAITING_APPROVAL`, `outcome=PARTIAL`, `gate=needs_review` — **not** `TERMINATED`.
- `write_review_suggestion()` runs **before** `finalize_terminal_artifacts()` so `review_suggestion.json` is on disk and listed in the manifest with a correct sha256.
- InternLoop appends `waiting_review` to `trace.jsonl` **after** review gating (outcome downgraded from `succeeded` to `partial`); `loop_trace.jsonl` must not claim `terminated/succeeded` for patch runs awaiting review.
- After human `approve`: direct-finalize to `TERMINATED` / `SUCCEEDED` / `gate=pass` (unchanged from P0-2).

**`report_path` priority**

- File-name priority: `report.md` → loop-specific reports (`development-report.md`, etc.) via `REPORT_NAME_PRIORITY`.
- Manifest fallback: only entries with `kind == "report"`; never `diff-summary.md`.

**Deferred review semantics**

- `review defer --until <date>` sets `status=deferred` and `deferred_until`.
- `ReviewStore.upsert_pending()` (called by `sync_from_runs`) must **not** reset a future-deferred item to `pending`.
- `approved` / `rejected` / `cancelled` decisions are terminal — sync never reopens them.
- Deferred items with `deferred_until > today` are hidden from default `review list`; they resurface when the date is due.

### Phase 2 — Define one failure-artifact contract

Every accepted run request must create a RunRecord and a run directory before risky work starts. Terminal outcomes include `SUCCEEDED`, `PARTIAL`, `BLOCKED`, `INVALID_INPUT`, `FAILED`, `INTERRUPTED`, and `CANCELLED`.

Every terminal path must contain:

```text
run_meta.json
gate_result.json
tool-results.json
report.md
artifact-manifest.json
loop_trace.jsonl
```

An empty `tool-results.json` is valid only when it explains that no tool call began. The manifest must list every present artifact and its checksum. **`artifact-manifest.json` is written exactly once** by `finalize_terminal_artifacts()` (atomic `.tmp` + replace); it must not list itself; sha256/size_bytes are always recomputed from final on-disk files. Individual loops may write other artifacts but must not overwrite the manifest after finalization. CLI errors may return non-zero, but must not lose the run ID or emit an unhandled traceback for an expected operational failure.

Test at least: missing/corrupt fixture, missing workspace/repository, command-not-found, pytest failure/timeout, git failure, ToolBroker rejection, Adapter block, malformed source, lock conflict, SQLite write error, and injected interruption.

### Phase 3 — Repair schema history with migration v4

Never edit the meaning of an applied v3 migration.

- retain v3 as the shipped `summaries` migration;
- add idempotent v4 to create/repair `review_items`, review decision events, required queue extensions, summaries, and missing indexes;
- make `db verify` inspect actual tables, columns, indexes, constraints, and continuous migration history—not only `MAX(version)`;
- when version 3 lacks review capability, print: `Run loop-pilot db migrate to apply v4 repair migration.`

Migration matrix:

```text
empty -> latest
v1 -> latest
v2 -> latest
v3 summaries-only -> v4 repaired
v3 review-only historical fixture -> v4 repaired
missing table / corrupt migration history -> verify fails safely
latest -> migrate again (no-op/idempotent)
```

Dry-run invariants apply to every row in the matrix: no database creation, schema mutation, migration row, journal/WAL residue, or target-file metadata change.

### Phase 4 — Make recovery real and safe

- persist the exact completed/current round at every checkpoint;
- do not reset consumed round, time, tool-call, or token budgets on resume;
- make repeated resume idempotent;
- reject resume after cancel/reject;
- revalidate Git HEAD, workspace identity, config hash, policy and adapter permission before continuing;
- permit automatic resume only from explicitly safe phases;
- require review for ACT/tool/adapter interruptions where side effects may be ambiguous.

Default CI must include the deterministic recovery suite. Separate subprocess kill tests may use a slower CI job, but may not be permanently hidden.

### Phase 5 — Complete 0.4-c review decisions

Minimum commands:

```text
review list
review show <id>
approve <id>
reject <id> --reason ...
defer <id> --until ...
cancel <id> --reason ...
resume <run-id>
```

Required behavior:

- `review_items` persistence and append-only decision events;
- `review_required.md` and `gate_result.json` for review-gated output;
- workers cannot approve themselves;
- patch-producing runs cannot become completed before a human decision;
- **direct-finalize approve**: `approve` on `patch.diff` runs sets `approved` + `TERMINATED` + `SUCCEEDED` + `gate_result.json=pass` without `resume_requested`;
- summary must not label `PARTIAL` / `WAITING_APPROVAL` / `needs_review` runs as completed.

## 4. Ideas Acceptance after Milestone A

### InternLoop controlled-real E2E

Use a temporary real Git repository and worktree with a committed Python bug and failing test. Require fail -> patch -> pytest pass -> real `git diff` -> pending review. The report and audit must contain the actual command output and patch checksum.

### PaperLoop evidence E2E

Use recorded source dossiers plus optional live retrieval. Test supported, irrelevant, over-strong, and missing-citation claims. Every decision must emit `claim_id`, source identity, evidence location, BibTeX key, support level, decision, and reason. A BibTeX entry alone is never evidence.

### DailyNews replay and live E2E

Default CI uses licensed/frozen GitHub, arXiv and RSS replay data for parsing, canonicalization, deduplication, explainable ranking, candidate routing, Inbox import, and summary. Optional `pytest -m live --run-live` validates small real source profiles without making normal CI network-dependent.

### Seven-day dogfood gate

Run for seven consecutive days with at least one real Inbox item, one review decision, one daily summary, and representative Intern/Paper/DailyNews work. Produce a weekly summary of completion rate, false positives, blocked reasons, recovery events, elapsed time, and manual corrections.
