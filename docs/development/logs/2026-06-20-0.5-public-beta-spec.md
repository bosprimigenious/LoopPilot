# 规划日志：0.5 Public Beta / PyPI 规格确立

**日期**：2026-06-20  
**决策类型**：版本边界与发布规格  
**状态**：已采纳（文档 only，未实现代码）

## 背景

0.4（recovery-and-automation）聚焦「维护者本人每日稳定运行」：SQLite、checkpoint/recovery、approval、OS scheduler。  
此前 V1 混合文档常把 PyPI、开源 README、CONTRIBUTING 与 0.4 能力绑在一起，导致：

- 外部用户期望「clone 即可 pip 发布」，与 0.4 实际交付（editable install + 开发文档）不符
- 0.5 被简化为「发 PyPI」一行，缺少 init/demo、示例、release CI、协作文件等可验收清单
- Legacy 映射未明确 **V1 PyPI/开源 → 0.5**，而非 0.4

## 决策

### 1. 核心口径

| 版本 | 定义 |
|------|------|
| **0.4** | 你自己每天稳定用（SQLite、recovery、scheduler、approval） |
| **0.5** | 别人也能安装、运行、理解、贡献（Public Beta / PyPI Release） |

**0.5 核心不是堆功能**，而是：发布质量、安装体验、文档、安全边界、示例、开源协作。

### 2. 版本命名

- Git tag / GitHub Release：`0.5.0-public-beta` 或 `0.5.0-pypi-beta`
- PyPI 预发布：`0.5.0b1`（PEP 440 beta）

完整链：

```text
0.1.0-mini → 0.2.0-practical-mvp → 0.3.0-adapter-mvp
  → 0.4.0-recovery-and-automation → 0.5.0-public-beta → 1.0.0-stable
```

### 3. 0.5 必须实现（写入 34 §6）

1. PyPI `pip install loop-pilot`，5–10 分钟跑通 demo
2. `loop-pilot init` / `loop-pilot init demo`
3. GitHub Actions：`ci.yml` + `release.yml`（Trusted Publisher 优先）
4. 开源 README（英文主体 + `docs/zh/`）
5. `examples/` 四套 demo
6. 用户文档：getting-started、concepts、configuration、safety、adapters、loops/*
7. CONTRIBUTING、SECURITY、CODE_OF_CONDUCT、LICENSE、CHANGELOG
8. 包发布质量检查清单（build、twine check、pipx install、无 outputs/memory/.env）

### 4. 0.5 明确不做

商业化、云服务、多用户、Web UI、插件市场、企业权限、自动 PR/deploy。

### 5. Legacy 映射修正

| 旧 V1 能力 | 新编号 |
|------------|--------|
| SQLite / recovery / scheduler | **0.4** |
| PyPI / 开源 README / CONTRIBUTING / examples | **0.5** |

### 6. 完成度估计

- 到 0.5：**80–90%**（自项目起点）
- 若 0.4 已完成，到 0.5 还差 **15–25%**（主要是打包、文档、示例、release，约 1–2 周）

### 7. approval CLI 稳定性

`approve/reject/resume/cancel`：若 0.4 未完全稳定，0.5 文档与 `--help` 标 **experimental**，不阻塞 PyPI beta 发布。
