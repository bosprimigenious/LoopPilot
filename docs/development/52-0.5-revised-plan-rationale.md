# 0.5 Revised Plan — Rationale

> **Date:** 2026-06-21  
> **Status:** Adopted (documentation)  
> **Spec:** [50-personal-daily-loop-0.5-spec.md](50-personal-daily-loop-0.5-spec.md)  
> **Decision log:** [logs/2026-06-21-0.5-plan-revision.md](logs/2026-06-21-0.5-plan-revision.md)

---

## 1. What changed

The 0.5 direction remains **valid** (safe personal autonomy: real schedule + unattended daily), but the **implementation order** and **scope** were revised after Truthful 0.4 acceptance work exposed blockers.

| Topic | Prior assumption | Revised decision |
|-------|------------------|------------------|
| Start 0.5 | After 0.4-d features exist | After **0.4-c** + **Truthful 0.4** Milestone A |
| First 0.5 code | Schedule install or `init personal` | **SafetyGate v1** in `src/loop_pilot/safety/` |
| Unattended runtime | Optional daemon | **No daemon** — OS scheduler calls CLI |
| PyPI / public beta | Sometimes conflated with 0.5 | **Optional**; not blocking safe autonomy |
| 0.4-d READY | Implied full 0.4 | 0.4-d alone does **not** imply 0.4 or 0.5 ready |

---

## 2. Why 0.4-c is a blocker (Step 0)

### 2.1 Truthful acceptance chain

[logs/2026-06-21-truthful-0.4-acceptance.md](logs/2026-06-21-truthful-0.4-acceptance.md) demonstrated:

```text
0.4-c acceptance (NOT READY): 0/16 passed
0.4-d acceptance: blocked by 0.4-c prerequisite (exit 2)
```

Unattended safe daily **reads** review state (Needs Review in `daily-summary.md`) and **must not** bypass human decisions. Without 0.4-c:

- No `review list` / approve / reject / defer CLI
- No `gate_result.json` contract on review-queue runs
- Safe autonomy could mask missing human gate — violating core LoopPilot safety model

### 2.2 Stabilization gaps

[50-0.4-stabilization-and-truthful-acceptance.md](50-0.4-stabilization-and-truthful-acceptance.md) lists additional P0 items (migration matrix, failure-path artifacts, kill/crash tests). 0.5 builds on recovery semantics; shipping install/unattended before stabilization risks **silent data loss** or **false READY**.

**Rule:** Do not start 0.5 implementation acceptance until Step 0 (0.4-c + stabilization Phase 0–1) is green.

## 3. Why SafetyGate FIRST (Step 1)

### 3.1 Problem

0.4-d deliberately refused dangerous actions:

- `schedule install --yes` → defer to 0.5
- `run daily` without `--dry-run` → refuse

0.5 introduces **real** OS mutation and **non-interactive** runs. Without a central gate:

- Schedule install could write crontab before policy is loaded
- Unattended path could drift into adapter calls via config mistake
- Audit trail would be fragmented across CLI modules

### 3.2 SafetyGate v1 role

Single evaluation point for **all** dangerous actions before side effects:

```text
CLI / Scheduler ──► SafetyGate.evaluate(action) ──► allow │ deny │ defer
                              │
                              ▼
                        audit record (JSON)
```

**First code in 0.5** must be `src/loop_pilot/safety/`, not scheduler install. Schedule and unattended commands **call** the gate; they do not embed policy checks ad hoc.

### 3.3 Alignment with existing layers

| Layer | Role |
|-------|------|
| ToolBroker (0.3) | Tool-level policy during loops |
| Review Layer (0.4-c) | Human decisions on run outcomes |
| SafetyGate (0.5) | **Autonomy-level** policy for schedule + unattended orchestration |

SafetyGate does not replace ToolBroker or review; it **composes** with them.

---

## 4. Why schedule install before unattended (Steps 2 → 3)

1. **Smaller blast radius** — install is one-time (or rare) OS write; daily run is recurring.
2. **Preview already exists** — 0.4-d `schedule install --dry-run` produces diff artifact; `--yes` extends with gate + idempotent install.
3. **Scheduler contract** — install must embed the exact CLI invocation (`run daily --unattended --safe`). Defining install first fixes the command string before unattended logic ships.
4. **Rollback story** — uninstall / dry-run diff is easier to test before unattended runs depend on the entry.

---

## 5. Why no daemon for 0.5

### 5.1 Rejected approach

A background `loop-pilot daemon` that polls inbox, wakes loops, and manages schedule internally.

**Problems:**

- Second process model (supervision, upgrade, crash restart)
- Duplicates OS scheduler responsibility
- Harder to audit — who triggered the run?
- Conflicts with personal-first scope: one maintainer, one machine, cron/Task Scheduler already present

### 5.2 Adopted approach

```text
OS Scheduler (cron / systemd timer / Task Scheduler)
        │
        ▼
loop-pilot run daily --unattended --safe
        │
        ▼
SafetyGate + verify + recovery-scan + summary (+ safe loops per policy)
```

**Benefits:**

- Standard ops tooling (systemd/cron logs)
- No extra port/socket/supervisor
- Clear attribution: `operator=scheduler` in gate audit
- Matches 0.4-d mental model (CLI-first)

Daemon may be reconsidered in 0.6+ for long-running adapters; **not** a 0.5 requirement.

## 6. Sub-milestone split (0.5-a / b / c / d)

| ID | Rationale for separate gate |
|----|----------------------------|
| **0.5-a** | Proves gate + install without unattended execution risk |
| **0.5-b** | Proves unattended safe path without crash matrix noise |
| **0.5-c** | Proves resilience after happy path exists |
| **0.5-d** | Human dogfood — 5 workdays — catches config/timezone/review friction |

Each sub-milestone has independent acceptance in [53-0.5-acceptance.md](53-0.5-acceptance.md).

## 7. What 0.5-prep allows (parallel during 0.4-c)

While 0.4-c is in progress, **documentation and non-production stubs** are OK:

- SafetyGate type definitions and interface tests (no real install)
- Schedule print / dry-run polish
- `status` read-only extensions
- Log schema design (`daily_run.jsonl`, gate audit)

**Not OK in prep:**

- Running `schedule install --yes` on maintainer production config
- Running unattended without `--safe`
- Declaring 0.5 READY in README/CHANGELOG

---

## 8. Relationship to Personal Beta (roadmap §7)

[34-version-roadmap-0x.md §7](34-version-roadmap-0x.md) still describes 0.5 as **Personal Beta** (`init personal`, config migration, optional PyPI).

**This revision narrows implementation priority:**

1. **Must-have for 0.5:** Safe autonomy (this spec)
2. **Optional in 0.5:** `init personal`, PyPI `0.5.0b1`
3. **Deferred to 0.8:** Public onboarding, CONTRIBUTING pack

Safe autonomy is the **blocking** definition of 0.5 done; PyPI is convenience, not proof of safety.

---

## 9. Alternatives considered

| Alternative | Why rejected |
|-------------|--------------|
| Ship 0.5 with 0.4-c stub (empty Needs Review) | Violates truthful acceptance; human gate missing |
| Daemon-first autonomy | Operational complexity; duplicates OS scheduler |
| Install + unattended in one PR | Hard to test; gate bypass risk |
| Skip SafetyGate; config flags only | No unified audit; easy to misconfigure |
| Full unattended with real adapters in 0.5 | Needs evaluation gate (0.6+); out of safe profile |
